*   **Problem:** 문서 작성에 과도한 시간 소요 (최대 1개월). 빠른 검증이 필요한 프로젝트에는 비효율적.
*   **Try:** 프로젝트 성격(MVP vs 정식)에 따라 문서화 수준을 유연하게 조절하는 전략 필요.

#### 🚓 DNA+ (폴리스드론)
*   **Action:** 이미지 회전 문제 수정 및 코드 경량화.
*   **Review:** 코드가 간결하나 확장성 부족. 동기식 API 통합 과정에서 제외 결정. *Status: Archived.*

#### 💬 Aloha (대화형 AI)
*   **Action:** 번역 엔진 교체 (Papago → ChatGPT)로 비용 구조 혁신.
*   **Problem:** 레거시 코드 누적 및 모델 버전 파편화. 비즈니스 로직과 LLM 모듈의 강한 결합도 문제 확인.

#### ⚙️ 서버 및 인프라
*   **성과:** 모니터링 시스템 도입으로 장애(NAS 접속 불가, 전원 장애) 원인 신속 파악.
*   **Lesson:** 서버실 전원 용량 예측을 통한 멀티탭 사전 증설 성과 (서울대 서버 이전 시 안정성 확보).

### 2분기 (Q2)
**집중 분야:** 프로토타이핑 및 신규 플랫폼 구축

#### 🤖 LLM 서빙 플랫폼 구축 (폐쇄망 대응)
*   **Keep:** Open WebUI, vLLM 등을 활용해 '동작하는 프로토타입'을 빠르게 구현 및 시연 성공.
*   **Problem:** 시연 이후 구체적인 사업화(Biz-Model)나 고도화 단계로 나아가지 못하고 정체됨.
*   **Try:** 단순 구현을 넘어, MVP의 기능과 제공 가치를 명확히 정의하고 후속 액션을 수립해야 함.

#### 🎬 미디어 문화향유 (웹 서비스 개발)
*   **Keep:** **2주 단위 스프린트** 도입으로 개발 속도 감각 유지.
*   **Problem:** 프론트엔드-백엔드 API 설계 합의 지연.
*   **Try:** '완벽한 시스템'보다 '스프린트 목표 달성'을 최우선으로 하여 딜레이 방지.

#### ⚡ 숏폼 자동 생성 (ShotsAI)
*   **Keep:** 매주 전략 회의를 통해 목표 공유.
*   **Problem:** 기간 내 MVP 최소 요구사항 미달성 (일정 지연).
*   **Try:** 유사 프로젝트 코드 재활용률을 높여 개발 속도 단축 (Build vs Buy/Reuse).

### 3분기 (Q3)
**집중 분야:** 프로젝트 고도화 및 아키텍처 개선

#### 🏛️ 미디어 문화향유 (NIPA 마무리)
*   **Keep:** **사이드카 패턴(Sidecar Pattern)** 도입. 비동기 AI 작업과 메인 프로세스의 의존성을 완벽히 분리하여 안정성 확보.
*   **Problem:** GPU 리소스 산정 실패로 2주 배포 지연 (로컬 환경 vs 프로덕션 차이 간과).
*   **Try:** 프로젝트 착수 전 **Capacity Planning(하드웨어 요구량 산정)** 절차 필수화.
*   **Insight:** "AI 코드는 관리 가능한 크기가 한정적이다. 의존성을 모듈 단위로 끊어내는 전략이 필수적."

#### ☁️ 네이버 마이박스 (비동기 아키텍처)
*   **Keep:** **Kafka 기반 비동기 시스템** 성공적 구축. 아키텍처 설계 → 로컬 검증 → 알파 → 배포의 정석적인 프로세스 이행.
*   **Problem:** 1인 프로젝트 체제로 인해 PM/협업 역량 성장이 제한됨.
*   **Try:** 브랜칭 전략 표준화 및 Jira/Issues 도입으로 협업 가능한 구조 마련.

#### 📺 SBS API 프로젝트 (임시)
*   **Failure Analysis:** AI(kiro)를 통해 비동기 모듈(Airflow) 도입을 시도했으나, **'동작하지 않는 그럴듯한 코드'** 생성으로 실패.
*   **Lesson:** "AI에게 맡기더라도 작업자가 전체 구조와 공식 문서를 이해하고 있어야 디버깅이 가능하다(Whitebox Approach)."

### 4분기 (Q4)
**집중 분야:** 프로덕션 배포 및 설치형 패키징

#### 💧 수자원공사 플럭 검출 (K-water)
*   **Achievement:** AI 협업을 통해 **통상 1주 소요되던 모듈 개발을 단일 프롬프트로 단축**. 전체 프로젝트 1주 내 완료 및 설치형 패키징 성공.
*   **Reflection:** "얼마나 빨리 했냐(Speed)"보다 **"얼마나 멀리 내다봤냐(Strategy)"**가 더 중요한 역량임. 아낀 시간을 더 큰 설계에 투자해야 함.

#### 🔄 네이버 마이박스 (운영 최적화)
*   **Action:** 실제 트래픽(일일 요청량, 이미지 용량)을 기반으로 디스크/리텐션 정책 수립.
*   **Result:** GPU Over-provisioning 해소를 통한 **비용 최적화 달성**.
*   **Collaboration:** 담당자(규영님)와의 커뮤니케이션 효율 향상.

## 🎯 2024년 총평 및 핵심 인사이트 (Unified Insights)

### 1. AI와의 협업: '가능성'과 '프로덕션'의 차이
*   AI로 새로운 모듈을 뚝딱 만드는 것은 가능(수자원, 숏폼)하지만, **'프로덕션 레벨'**의 기능(재시도, 로깅, 추적)을 갖추는 것은 별개의 영역입니다.
*   **Build vs Buy:** 복잡한 기능은 AI에게 코딩을 시키기보다 검증된 솔루션(Airflow, Celery)을 도입하고 설정을 맡기는 것이 훨씬 효율적입니다.

### 2. 아키텍처 역량: 의존성 관리
*   **사이드카 패턴**의 성공 사례처럼, AI가 생성한 코드의 '블랙박스 문제'를 해결하는 열쇠는 **철저한 모듈화와 의존성 분리**에 있습니다.

### 3. 성장 방향성: '속도'에서 '깊이'로
*   수자원 프로젝트를 통해 '압도적인 개발 속도'는 검증되었습니다. 이제는 확보된 시간을 활용해 시스템의 전체 그림을 그리는 **'아키텍처 설계'와 '운영 전략(Planning)'** 역량을 키우는 데 집중해야 합니다.
