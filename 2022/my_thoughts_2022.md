# My Thoughts - 2022

## Q1 Reflections

### Key Learnings

#### Lesson 1: Less is More
**깨달음**: 모든 기능을 전부 제공하는 것이 항상 정답은 아니다.
- 불규칙적인 분석 요청 시나리오에서는 배치 단위 처리가 항상 효율적이지 않을 수 있음
- 요청이 많지 않은 상황에서는 복잡한 최적화가 오히려 노력 대비 비효율적일 수 있음

#### Lesson 2: Design for Change
**깨달음**: 초반에 설계한 사양은 반드시 바뀐다. 변화를 대비하여 설계해야 한다.
- 프로젝트 초기 설계 시 유연성 확보가 중요
- 고정된 사양보다는 변화 가능성을 고려한 설계 필요

#### Lesson 3: Python의 본질 이해
**깨달음**: Python은 인터프리터 싱글 쓰레드 언어이다. 기능을 설계할 때는 이러한 성질을 고려해야 한다.
- Python의 GIL(Global Interpreter Lock) 특성 이해
- Multi-processing vs Multi-threading 선택의 중요성
- 언어 특성에 맞는 아키텍처 설계 필요성

### Strategic Direction Shift
**기존 접근**: 프로젝트를 빨리 끝내는 것에 집중
**새로운 방향**: 안정적으로 설계하는 것을 우선시

**이유**:
- 기존 시스템의 안정성 부족 문제 인식
- 급한 일정 때문에 다루지 못했던 부분들 보완 필요
- 과거 경험을 기반으로 모니터링 지속 필요

**새로운 우선순위**:
1. 시스템 안정성
2. 단순화

## Q2 Reflections

### Key Learnings

#### Lesson 4: 구조적 결함은 나중에 발견된다
**깨달음**: 코딩 단계에서는 확인하기 힘든 프로그램 구조적 결함이 언제나 있을 수 있다.

**새로운 접근법**:
1. 일단 기간에 맞춰 동작하는 코드를 만든다
2. 그 다음에 사용하고 있는 기능들이 정확히 어떻게 동작하는지 파악한다
3. 무엇을 유의해야 하는지 학습한다

**실제 경험**:
- DNA+ 프로젝트: 모델이 느린 환경에서는 어떠한 최적화도 한계가 있음
- 시스템 성능 개선 목표 설정 시 모델 실행 시간과 요구되는 스펙을 비교해야 함
- 모델이 낼 수 있는 성능의 100%를 내고 있는지 측정이 중요

#### Lesson 5: 병목현상은 예상치 못한 곳에서 발생한다
**이미지 스케줄링 문제 경험**:
- 두 프로세스 간의 병목 현상 최소화를 위해서는 두 시스템의 타이밍을 계산할 수 있어야 함
- 각 시스템을 단편적으로 바라보고 설계 진행 시 예상치 못한 곳에서 새로운 형태의 병목현상 발생
- 전체적인 시스템 관점에서의 설계가 중요

### Architecture Design Insights

#### Message Queue Architecture 도입
**문제 인식**:
- 복잡한 AI 분석 시스템의 인터페이스 복잡도
- GPU 리소스의 효율적 사용 필요

**해결 과정**:
- Message Queue (RabbitMQ) 기반 아키텍처 설계
- 동기식 API와 비동기 처리의 하이브리드 접근
- 리소스 제한에 따른 요청 수 관리 메커니즘 구현

**성과**:
- 인터페이스 복잡도 3배 개선
- GPU 사용 효율 향상
- Fail 시나리오 체계적 관리
